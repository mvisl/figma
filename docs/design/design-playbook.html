Design Playbook

Last updated: 2026-06-21. This file is the shared source of truth for design and Figma work across local Codex workspaces and the GitHub mirror.

Keep this page synchronized with the GitHub copy. Chats started from a local workspace should read the local file; work on another computer should pull or copy the latest GitHub version before starting.

How To Use This File

  • Start with General Rules for any design, Figma, or UI task.
  • Then open the matching project section, such as Oboard or Libertex.
  • When a decision is made in chat, Figma comments, video feedback, or product review, add it to the relevant section and changelog.
  • If a project-specific rule conflicts with a general rule, follow the project-specific rule and record why.

General Rules

Source Priority

  1. Use the project's primary design system first.
  2. If the component or pattern is not in the design system, use the existing live product UI.
  3. If the live product does not cover the case, use existing Figma pages and previous project screens.
  4. Use secondary design systems only for the explicitly allowed properties in the project section.
  5. Create a new element only when no suitable existing pattern exists, and document the reason.

Figma Workflow

  • Work from the current iteration and the user's annotated frames, not from old exploration frames unless explicitly told.
  • Preserve comments and use them as requirements until the user marks them obsolete.
  • When video feedback exists, treat the transcript and screenshots as clarification for the Figma comments.
  • When a detail is ambiguous, request a screenshot or timestamp instead of guessing a visual state.
  • Reuse existing components, instances, variables, color styles, and text styles whenever available.

Style Rules

  • Colors and text styles must come from the project's primary design system unless a project section says otherwise.
  • Spacing, radius, and sizing may come from a secondary system only when the project section allows it.
  • Do not invent a new visual language when the product already has the same pattern elsewhere.
  • Before finalizing UI work, verify that text fits, controls match existing product behavior, and states are accounted for.

Decision Logging

  • Record decisions as short, dated entries in the relevant project's changelog.
  • Keep the latest rule in the main project section; changelog explains how it changed.
  • Move stale or overridden decisions out of active rules and into notes if they still matter historically.

Design System Priority

  2. Live Oboard product: use existing pages in Chrome when a matching pattern already exists in production.
  3. Existing Figma pages: use elements from current or previous Oboard screens when they are not in the formal design system.
  4. Simple Design System: allowed only for spacing, dimensions, radius, and distance references.
  5. New local design: only when none of the above contains the needed pattern.

Dashboard / Collection Model

  • A collection is now a label/group/tag, not an ownership or permission container.
  • No user owns a collection.
  • Any user can add a collection label to any dashboard where they can perform the dashboard-level action.
  • A dashboard can belong to multiple collections at the same time.
  • Dashboard permissions remain on each dashboard.
  • A collection can be visible while some dashboards inside it remain private to the current user.

Dashboard / Collection Views

View Expected Behavior
Dashboards Shows the current dashboard list. Add collection membership controls, likely as a column or dropdown similar to the production label/collection selector.
Collections Shows a list of collections. Each collection expands to show dashboards inside it.

Controls And Actions

  • Add a top switcher between Dashboards and Collections, based on the existing Oboard segmented switcher pattern from the Objectives alignment progress page.
  • Collection actions: Edit collection and Delete collection.
  • Collection share can exist as a direct collection-level action if confirmed, but collection permissions are not the source of dashboard access.
  • Do not show Owner or Public/Private fields for collections.
  • Dashboard actions remain the existing dashboard actions, with collection assignment added.
  • The collection assignment dropdown should support search and add/create behavior like the current production pattern.
  • Drag and drop is not needed because dashboard-to-collection membership is many-to-many.

Private Dashboards Inside Collections

  • If a collection contains a dashboard the current user cannot access, show a private dashboard placeholder.
  • The placeholder should explain that this is a private dashboard.
  • Include a Request access action.
  • Do not use the old collection-level permission banner for this case.
